# P1.3 Availability Windows — Manual QA Checklist

Covers PR-L/PR-M (pure diff engine + apply/cancel/modify wrappers),
PR-N (REST routes + preview endpoint), PR-O (rollback via
`ai_proposals.kind='availability_change'`), PR-P (availability page
CRUD + review modal), PR-Q (Today chip, WeeklyStrip overlay, /you
entry card), PR-R (API boundary hardening: length cap, immutable kind
rejection, drift messaging).

Prereqs:
- Logged in user with `profiles.timezone` set (try `America/Los_Angeles`,
  `Asia/Tokyo`, `Europe/London` at least once each — crossing midnight UTC
  is where TZ bugs hide).
- At least one active phase with a weekly pattern (any training day
  codes) and a seeded `calendar_events` row per day_code — so roll-forward
  has something to insert into.
- 10–20 `plans` rows in (today-7, today+21]: mix of `planned/template`,
  `planned/manual`, `planned/ai_proposed`, and one `done` — so preservation
  rules are observable.
- No other pending proposals (or a known-benign set) — availability
  changes fire `reconcile()` which can race with phase-transition
  detection.

## Create — happy path

- [ ] `/you/availability` with zero windows shows empty state and a
      single `+ Declare a window` CTA.
- [ ] Open the form, pick kind=travel, today .. today+6, strategy=auto,
      empty note. Tap "Review changes".
- [ ] Preview returns a diff with `creates`, `rows_replaced` (template
      rows in the range), `rows_preserved_*` counts, and no `conflicts`.
      Confirm counters match what's actually in `plans`.
- [ ] Tap "Apply" → banner shows "Applied. N days queued." Dialog
      closes. Window appears in the Active section.
- [ ] Inspect `availability_windows`: one row, `status='active'`,
      `created_at ≈ now`, `cancelled_at IS NULL`.
- [ ] Inspect `ai_proposals`: one row, `kind='availability_change'`,
      `status='applied'`, `applied_at` set, `diff.intent='create'`,
      `diff.window_after.id` matches the new window.
- [ ] Inspect `plans` in the window range: template rows replaced per
      strategy (auto picks bodyweight vs rest based on kind), existing
      `manual`/`ai_proposed`/`done` rows untouched.

## Create — validation surface

- [ ] Submit with `starts_on > ends_on` → 400, dialog shows the
      inline error, no network POST to apply.
- [ ] Submit with a range longer than 365 days → 400 with
      "window length N days exceeds max 365".
- [ ] Submit without picking a kind → form disables "Review changes".
- [ ] `note` over 500 characters is rejected server-side and client
      truncates in the textarea.
- [ ] Overlap with an existing active window → preview returns 409 with
      `conflicts` populated; dialog renders the conflict card and the
      "Apply" button is disabled.
- [ ] Overlap check clamps to today: an overlap entirely in the past
      against another past window does NOT block create (by design —
      past windows don't affect future plans).
- [ ] `kind='pause'` with `strategy='bodyweight'` is rejected server-
      side as an invalid combination (pause forces rest/suppress).

## Create — TZ edge cases

- [ ] User in `America/Los_Angeles` at 23:30 local creates a window
      `starts_on=today+0`. `buildCreateWindowDiff` must resolve `today`
      from `formatInTimeZone(new Date(), tz, 'yyyy-MM-dd')`, not UTC.
      Verify `diff.creates[0].date` is the user's local date, not the
      UTC date.
- [ ] Same test at 01:00 `Asia/Tokyo` with UTC still on yesterday —
      `today` should be the Tokyo date.
- [ ] Change `profiles.timezone` to `UTC`; next preview uses UTC today.
- [ ] `profiles.timezone` null → `tz='UTC'` fallback; no crash, no
      "Invalid time zone" error.
- [ ] Window crossing DST spring-forward in local TZ (e.g.
      `America/New_York`, March window containing the second Sunday):
      inclusive day count is correct; no missing day in `creates`.

## Modify — happy path

- [ ] Edit an active window: extend `ends_on` by 3 days. Preview shows
      additional `creates` for the 3 new days; existing window days
      untouched (`unchanged` count in diff).
- [ ] Apply → `availability_windows` row updated in place (same id),
      `ai_proposals` row has `diff.intent='modify'`, `window_before`
      has the old range, `window_after` the new range.
- [ ] Shrink `ends_on` by 2 days. Preview shows `rows_restored` for the
      dropped tail days (template rows come back). Apply → plans in the
      tail are re-inserted from template, not left blank.
- [ ] Change `strategy` only (auto → bodyweight): creates re-computed,
      `prescription.blocks` differ, no date-range change.
- [ ] Set `note` to `null` from a non-null value: stored as null; diff
      intent still `modify`.

## Modify — immutable kind

- [ ] PATCH `/api/availability/:id` with `{kind: 'injury'}` → 400 with
      `"kind is immutable — cancel this window and create a new one to
      change kind"`.
- [ ] POST `/api/availability/preview` with
      `{window_id, patch: {kind: 'injury'}}` → 400 with same message.
- [ ] UI: in edit mode, kind selector is disabled with a "locked"
      helper line; keyboard focus skips over it.

## Modify — validation surface

- [ ] Empty patch (`{}`) → 400 "at least one field required".
- [ ] Patch with `starts_on > ends_on` (from explicit bounds) → 400.
- [ ] Patch that flips only one bound so the resolved range becomes
      `starts_on > ends_on` → 400 from `applyModifyWindow`'s resolved-
      range check.
- [ ] Patch that stretches resolved range over 365 days → 400 "exceeds
      max 365".
- [ ] Patch that creates an overlap with another active window → 409
      with `conflicts`.
- [ ] Modify a window whose `status='cancelled'` → 409
      `window_not_active`.
- [ ] Modify a window that doesn't belong to the caller (other user) →
      404 `window_not_found` (RLS + explicit `.eq('user_id', user.id)`).

## Cancel — happy path

- [ ] Cancel an active window via the `× Cancel` button. Confirm dialog
      appears with a summary line ("3 days of bodyweight sessions will
      revert to your plan"). Two-step confirm.
- [ ] Apply → `availability_windows.status='cancelled'`, `cancelled_at`
      set. Template plans restored for any future dates in the window.
- [ ] Window moves from Active → Recent section on `/you/availability`.
- [ ] `ai_proposals` row has `diff.intent='cancel'`, `window_before`
      populated, `window_after=null`.

## Cancel — past-window edge cases

- [ ] Cancel a window entirely in the past (`ends_on < todayIso`).
      Confirm dialog shows "0 future days affected" and explicitly
      states plans are unchanged.
- [ ] Apply → row moves to `status='cancelled'`; `plans` untouched
      (past plans are frozen; cancel is a recordkeeping op).
- [ ] Cancel a window straddling today: only future-from-today plans
      are restored. Plans on days `< todayIso` stay as they were.
- [ ] Cancel a window on its last day (`ends_on = todayIso`): no plan
      restoration — today's plan stays (we don't rewrite today).

## Rollback — availability_change audit

- [ ] On `/you/availability` expand a window's history. Tap "Undo"
      on the most recent applied proposal.
- [ ] POST `/api/proposals/:id` with `{action: 'rollback'}` → 200, the
      original proposal's `status='rolled_back'`, a new
      `availability_change` row inserted with `diff.intent='rollback'`
      and `diff.rollback_of = <original id>`.
- [ ] Rolling back a `create` cancels the created window.
- [ ] Rolling back a `cancel` re-creates the window with a NEW id
      (old cancelled row stays). The new row's `id` appears in
      `diff.window_after`.
- [ ] Rolling back a `modify` applies the reverse modify: old range
      and strategy restored.
- [ ] Rolling back an already-rolled-back proposal → 400
      `rollback_already_rolled_back`.
- [ ] Attempting rollback on a `kind != 'availability_change'`
      proposal → 400 `rollback_target_not_availability`.
- [ ] Rollback UI: once a proposal is rolled back, the Undo button
      hides / disables on that audit row.

## Preview/Apply drift — race handling

- [ ] Open the edit dialog for window W in tab A. In tab B, cancel W.
      Back in tab A, tap "Review changes" → preview returns 409
      `window_not_active`. Dialog shows "This window was cancelled
      elsewhere. Refresh to see the latest state." Toast fires; form
      auto-closes; page refreshes to show W in Recent.
- [ ] Same scenario but W is deleted server-side (bypass RLS). Preview
      returns 404 `window_not_found`. Dialog shows "This window was
      just removed."
- [ ] Preview succeeds, user stalls on the review modal, another
      client modifies W. User taps Apply → PATCH returns 409 (because
      before-snapshot mismatch). Review dialog surfaces the error and
      offers Back to form.
- [ ] Two tabs each open the create dialog for overlapping ranges,
      preview OK, both tap Apply. The second apply gets 409
      `overlaps_existing` — pre-flight overlap check in
      `applyCreateWindow` catches it; no double-insert. (Known
      limitation: without a DB exclusion constraint, a nanosecond-
      tight race could still slip through. See
      `docs/p1.3-known-limitations.md`.)
- [ ] Idempotency: same apply POST issued twice (e.g. retry-on-network-
      blip). Second call returns 409 `window_not_active` (first call
      moved status to applied). UI treats as success and refreshes.
      (Follow-up: a `request_id` dedupe is filed under tech-debt.)

## WeeklyStrip overlay — Today page

- [ ] With an active window covering part of the current week, each
      day-dot in that range shows a small kind glyph in its top-right
      corner (travel → plane, injury → bandage, pause → dash-circle).
- [ ] Days outside the window have no badge.
- [ ] Aria-label on a windowed day reads `"Tue 2026-04-21 (travel
      window)"` — verifiable via screen reader or dev-tools inspector.
- [ ] Badge color matches kind tint; ring 1px `border` and
      `bg-panel` so it reads against both empty and filled dots.
- [ ] Toggling between a soft-hero week and a week with an injury
      window does not leave stale badges; strips re-mount cleanly on
      route change.
- [ ] Zero active windows → zero badges; no extra DOM nodes.

## Active window chip — Today page

- [ ] An active window whose range contains today renders an
      `ActiveWindowChip` between `WeeklyStrip` and the banners, with
      tinted bg/border matching kind.
- [ ] Chip text reads, e.g., "Travel window · ends Sat" for a window
      ending in ≤ 7 days, or "Travel window · ends 2026-05-01" for
      later.
- [ ] Chip is a `<Link>` to `/you/availability`; keyboard-focusable;
      focus ring uses static Tailwind class `ring-accent/40` (verify
      focus outline is visible, not invisible due to dynamic-class
      bug).
- [ ] Queued-but-not-active windows do NOT render the chip. Only
      `starts_on <= todayIso <= ends_on` qualifies.
- [ ] Two active windows overlapping today (should be impossible via
      validation but possible in seed data) → chip renders the first
      by `starts_on`; no crash.

## /you entry card — availability summary

- [ ] Zero active windows → card shows "No windows declared" with a
      link to `/you/availability`.
- [ ] One active-now window → card shows kind chip + "through Sat" (or
      date) + strategy label ("Bodyweight", "Rest days", "Auto-paused").
- [ ] No active-now window but N queued → "N queued" link copy.
- [ ] Active + queued both present → renders active-now block with an
      "+ N queued" tail.
- [ ] Card is server-rendered from `page.tsx`; no client-side fetch on
      load; no layout shift after hydration.
- [ ] `profiles.timezone` null → card's "through Sat" falls back to
      UTC-derived relative phrase; no crash.

## Reconcile coupling

- [ ] After any successful create/modify/cancel, `reconcile()` fires
      fire-and-forget with cause `'availability_changed'`. Verify via
      server log: exactly one reconcile call per apply.
- [ ] If reconcile errors out, the API call still returns 200 (apply
      was already committed). Error is logged, not surfaced to the
      client.
- [ ] A failing reconcile does not leave the `availability_windows` or
      `ai_proposals` row in a half-applied state (they're written
      atomically in the apply wrapper; reconcile is downstream).

## Auth / RLS

- [ ] `GET`/`PATCH`/`DELETE` `/api/availability/:id` with another
      user's window id → 404 `window_not_found`.
- [ ] Same for `/api/availability/preview` with a foreign `window_id`
      → 404.
- [ ] Unauthenticated (no session) → every availability route returns
      401.
- [ ] SSR `/you/availability` without a session redirects to `/login`.

## Regression sweep

- [ ] Phase-transition banner/hero still renders when a soft/hard PT
      proposal is pending on Today — availability banner stacks above
      pending-banner but below the hero per existing ordering.
- [ ] Adjust proposals still apply normally; `action=rollback` on a
      non-availability proposal → 400.
- [ ] `/today` weekly strip with zero active windows renders
      identically to pre-PR-Q baseline (no blank badge slot, no extra
      CSS class leaks).
- [ ] Existing tests in `src/lib/availability/diff.test.ts` and
      `src/lib/availability/apply.pure.test.ts` still pass
      (`npm run typecheck && npx tsx --test src/**/*.test.ts`).

## Known limitations (not blocking P1.3)

- No DB exclusion constraint on overlapping active windows. Pre-flight
  check is correct for single-client use and covers the common case
  but does not guarantee invariant under concurrent inserts. Tracked:
  `docs/p1.3-known-limitations.md` → "overlap invariant".
- No `request_id` dedupe on apply. A double-submit during a network
  blip works correctly (second call gets 409) but the error message
  is slightly confusing. Tracked: `docs/p1.3-known-limitations.md`
  → "idempotency".
- Preview/apply can drift if the user stalls on the review modal.
  Drift is caught by the before-snapshot check on apply, but the
  error UX falls back to a generic message. Follow-up is to surface
  a targeted "Window was modified — re-preview" CTA in the review
  dialog.
