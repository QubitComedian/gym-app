# P1.2 Phase Transitions — Manual QA Checklist

Covers PR-I (pure diff engine), PR-J (reconciler detection + apply wrapper + API routing), PR-K (soft banner + hard hero + Today integration).

Prereqs:
- Logged in, at least one active phase with a `target_ends_on`.
- A weekly template bound to the active phase (non-empty `weekly_patterns`).
- For full transition coverage: a second phase queued with `ordinal > active.ordinal`, its own weekly pattern, and at least one `calendar_events.day_code` bound to it.
- A handful of `plans` rows in (today, today+21] — mix of `planned/template`, `planned/manual`, `planned/ai_proposed`, and one `done` — so preservation rules can be observed.

## Detection — reconciler

- [ ] No proposal fires when `target_ends_on` is > 7 days out. Bump `reconcile({cause:'activity_logged'})` and confirm `phase_transition_detected === false`.
- [ ] Soft-tier proposal fires when `target_ends_on ∈ [today+1, today+7]`. `ai_proposals.kind === 'phase_transition'`, `triggered_by === 'phase_ending_soon'`, `diff.tier === 'soft'`.
- [ ] Hard-tier fires when `target_ends_on <= today`. `triggered_by === 'phase_ended'`, `diff.tier === 'hard'`.
- [ ] Open-ended phase (null `target_ends_on`) produces no proposal.
- [ ] Phase with `status !== 'active'` produces no proposal (both `classifyPhaseTransition` and the reconciler gate).
- [ ] No second proposal is inserted while one is pending for the same phase — re-run reconcile, confirm single pending row in `ai_proposals`.
- [ ] Extend the active phase's `target_ends_on` after a pending proposal exists. Next reconcile: old pending row is auto-rejected ("Phase-transition proposal superseded — phase target changed."), a fresh proposal fires with the new target.
- [ ] Active phase whose `status` has drifted but whose date range still contains today is detected via the fallback date-range match in `transition.ts`.
- [ ] `last_reconciled_at` freshness still applies — chatty reloads don't spam proposals.

## Detection — diff shape

- [ ] Default option is `transition` when next phase exists AND its pattern has at least one bound training day.
- [ ] Default option is `extend_2w` when no viable next phase (no pattern, no bound events, or no next phase at all).
- [ ] `end_phase` option is present **only** when there is no queued next phase.
- [ ] `reassess` option is always present.
- [ ] Each `extend_{1,2,4}w` option has `phase_updates[0].patch.target_ends_on` equal to `today + 7/14/28`, and `creates` fill missing dates from the OLD pattern.
- [ ] `transition` option has TWO phase updates: old → `completed/target_ends_on=today/actual_ends_on=today`, new → `active/starts_on=today+1`.
- [ ] Orphan slots show up in `summary.orphan_day_codes` (not silently hidden).
- [ ] Preservation: `status !== 'planned'` rows, `source = 'manual'`, `source = 'ai_proposed'`, and rows claimed by a different phase are all left untouched and surfaced under `summary.skipped_*`.

## Soft banner UX (`PhaseTransitionBanner`)

- [ ] Stacks above `PendingBanner` on `/today` when the pending proposal is soft-tier.
- [ ] Eyebrow reads "Phase ending today / tomorrow / in N days" based on `days_until`.
- [ ] Primary CTA label matches the default option's `cta_label` (e.g. "Start P3", "Extend 2w").
- [ ] One tap on the primary CTA POSTs `action=accept_option, option_id=<default>` and refreshes the page.
- [ ] Successful apply updates `ai_proposals.status` to `applied` and appends the chosen option to the rationale trail.
- [ ] "Decide later" hides the banner for the rest of the tab (sessionStorage `pt-dismiss:{id}`) AND POSTs `action=dismiss` so the row leaves `pending`.
- [ ] Tab reload after a "Decide later" dismiss does NOT re-show the banner (proposal is rejected server-side).
- [ ] Error surface: kill the dev server mid-click, CTA reports a human error string under the banner; banner re-enables for retry.

## Hard hero UX (`PhaseTransitionHero`)

- [ ] Replaces `TodayHero` (and `LastSessionCard`) on `/today` when the pending proposal is hard-tier.
- [ ] Eyebrow reads "Ended N days ago · <phase_code>" when overdue, "Ends today · <phase_code>" when due today.
- [ ] All options render as a radio group, keyboard-navigable via Tab + Space.
- [ ] Default option is pre-selected; selecting a different option updates the CTA label and clears the end-phase confirm state.
- [ ] Each option shows a compact chip line (e.g. "+9 new · −7 replaced · 2 orphans · 3 preserved"); lines are suppressed when the option is empty.
- [ ] "New target: YYYY-MM-DD" appears under extend options (not on `reassess` or transition/end).
- [ ] "Recommended" chip renders only on the default option.
- [ ] "Ends phase" chip renders only on `end_phase`.
- [ ] Accepting `reassess` redirects to `/check-in`, updates the proposal to `applied` with rationale note "User chose: reassess".
- [ ] Accepting `end_phase` requires TWO clicks — first tap flips the CTA to rose-tinted "Tap again to end phase"; a different option being selected resets the confirm.
- [ ] Accepting `transition` flips phase statuses (old → completed, new → active) and replaces replaceable planned rows in (today, today+21]. Confirm via SQL.
- [ ] Accepting `extend_2w` stretches `target_ends_on` and creates missing plans from the OLD pattern. Existing `planned` rows are left alone (no duplicates on the same date).
- [ ] After any successful apply, reconcile is fire-and-forget — Today re-renders with the new phase in the header and `status='applied'`.
- [ ] "Decide later" dismisses for the tab only; proposal stays `pending` so tomorrow's load re-opens the hero.
- [ ] Double-click protection: the CTA disables while a POST is in flight.

## Hero-conflict ordering

- [ ] Seed both a pending `return_from_gap` (hard tier) and `phase_transition` (hard tier). `/today` renders the RFG hero; PT is suppressed for this render.
- [ ] Accept the RFG hero → page refresh → reconcile picks PT back up (either as a fresh proposal or the same row, depending on whether RFG changed phase state). PT hero now shows.
- [ ] Seed both as banners (RFG soft + PT soft). Both stack above `PendingBanner`, in order: RFG first, PT second.
- [ ] Seed RFG hero + PT soft banner. Hero renders; PT banner also renders above the hero (banners always stack, heroes never overlap).

## API route

- [ ] `POST /api/proposals/:id` with `action=accept_option, option_id=<unknown>` on a phase_transition returns 400 with `error=unknown_option`.
- [ ] `action=accept_option` with missing `option_id` → 400.
- [ ] `action=apply` on a `phase_transition` proposal → 400 with "phase_transition requires action=accept_option + option_id".
- [ ] `action=accept_option` on an `adjust` proposal → 400 with the existing error.
- [ ] `action=reject` / `dismiss` on a `phase_transition` sets status to `rejected`; no plan mutations.
- [ ] Apply on an already-applied proposal → 400 with "already applied".
- [ ] Reassess redirect: response body includes `redirect: '/check-in'`; the hero / banner handles it client-side.
- [ ] Applied response includes `applied` counts and `skipped` counts; front-end does not rely on them but they're available for debugging.

## Apply correctness (SQL spot-checks)

- [ ] `applyPhaseTransition` on a transition: `phases.status` updated (two rows), planned template rows in window deleted, new phase_id rows inserted with `source='template'`.
- [ ] Plans with `status='planned'` + `source='ai_proposed'` are never deleted.
- [ ] Plans with `status='done'` are never deleted (guarded by the `.eq('status','planned')` in apply).
- [ ] Running the same apply twice (e.g. via a double-POST) does not double-insert — the second call either rejects on "already applied" or is a no-op (check `applied_at` monotonicity).
- [ ] Simulate partial failure by revoking RLS on `plans` mid-apply: phase_updates still land; next reconcile pass heals the missing plan rows via roll-forward.

## Regression sweep

- [ ] Pre-existing `return_from_gap` soft banner still renders and applies.
- [ ] Pre-existing `return_from_gap` hero still renders and applies.
- [ ] Adjust proposals still apply via the legacy `action=apply` path.
- [ ] `PendingBanner` no longer shows phase_transition rows (they're filtered out of `otherPending`).
- [ ] Today page loads with zero pending proposals render exactly as before.
